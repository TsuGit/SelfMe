Execution Plan

SelfMe Agent CLI Roadmap

当前只做一件事：先把一个 TypeScript 写的单体 agent CLI 做到非常完整、非常成熟， 目标是做到顶级 agent CLI 应有的完整度、稳定性和使用体验。 桌面端只作为后续可能性，前提是 CLI 已经足够成熟。

主路线： TypeScript 就是当前确认的技术栈。目标是把这条路线发挥到极致，做出顶级 agent CLI 应有的完整度、稳定性和体验。
Decision

为什么选择 TypeScript

  • 当前最重要的是把 agent workflow、task、approval、tool orchestration、session 体系整体做强。
  • TypeScript 更适合快速推进 session、task、approval、tool orchestration、provider integration。
  • 行业现实已经证明，优秀的 agent CLI 完全可以建立在 TS / Node.js 之上。
  • 后面如果真的需要桌面端，CLI 核心做成熟之后再包壳，成本更低。
路线要求： 选择 TypeScript 不是临时方案，也不是降级方案。终端层、editor、runtime、PTY 都要按产品级要求设计，并把这条技术路线做到最强。
Scope

当前范围

  • 只做 CLI。
  • 只做单体本地进程。
  • 直接连接模型 provider。
  • 直接执行本地工具。
  • 保存本地配置、transcript、logs、task state。

后续可能做

  • 桌面端。

但前提只有一个：CLI 已经非常完整、非常稳定、非常顺手。

Foundation

Terminal / Editor Foundation

这一层不是 UI 细节，也不是后期润色，而是整个 CLI 是否能成为产品的基础设施。 如果输入、光标、重绘、滚动、IME 处理做不好，后面 runtime、tools、planning 再强，整体体验也不会成立。

  • 输入区必须使用真实终端光标，不接受伪光标方案长期存在。
  • 消息区和输入区必须分离，输入区固定，消息区独立滚动。
  • 必须减少全屏粗暴重绘，逐步收敛到稳定区域重绘。
  • Editor 必须独立承担多行输入、cursor、history、粘贴和后续选择能力。
  • 中文输入法体验必须作为专门问题处理，不能假设终端会自动替我们解决。
执行要求： Terminal / Editor Foundation 属于前置阶段，不是最后优化阶段。必须在 agent 复杂能力继续膨胀之前，把输入体验和渲染基础做稳。
Contracts

Core Contracts

这一节不是补充说明，而是开工前必须先钉死的核心契约。 如果这些不先写清，后面最容易在 runtime、tools、storage、UI projection 上发生第二次返工。

1. Runtime Event Model

整个系统内部统一围绕事件流工作，UI 不直接拼业务逻辑，storage 也不直接依赖界面状态。

  • UserMessageSubmitted
  • AssistantStreamStarted
  • AssistantDeltaReceived
  • AssistantCompleted
  • ToolCallRequested
  • ToolExecutionStarted
  • ToolStdoutAppended
  • ToolStderrAppended
  • ToolExecutionCompleted
  • ApprovalRequested
  • ApprovalGranted
  • ApprovalDenied
  • TaskStateChanged
  • SessionCheckpointSaved
  • RuntimeErrorRaised

这些事件都应该带： eventId、 sessionId、 taskId?、 timestamp、 source、 payload。

2. Tool Contract

所有工具统一挂到 registry，接口不能随着工具种类增加而各写一套。

  • 每个工具必须定义 name、description、inputSchema、approvalPolicy。
  • 执行入口统一为 invoke(input, context)。
  • 返回值必须区分 summary、structuredOutput、rawLogs。
  • 长任务工具必须支持 cancel() 或 runtime 中断协议。
  • shell 类工具必须保留 stdout、stderr、exitCode。
  • 失败态不能只抛异常，必须能落成结构化结果，方便 transcript 和恢复。

第一版即使只有 shell 和 files，也要按完整协议写。 否则后面再加 search、planner、browser 时一定会散。

3. Context Strategy

顶级 agent CLI 不是“把全部历史硬塞进模型”，而是要管理 working context。

  • 最近对话保持完整。
  • 较早对话允许压缩成摘要块。
  • 工具原始日志默认不直接进入下一轮模型上下文，只进入 transcript / logs。
  • 工具结果进入上下文时优先使用 summary 和 structuredOutput。
  • 长会话必须支持 context compaction，而不是无限增长。
  • session history 和 model working context 必须是两个概念，不能混在一起。

第一版就要明确：用户看到的完整记录，不等于每一轮都传给模型的完整上下文。

4. Approval Policy

审批流必须是 runtime 的一等能力，不是工具层随手弹个确认框。

  • 每个工具都要声明默认审批策略：always、on-risk、never。
  • 文件写入、危险 shell、删除类操作必须能进入高风险审批。
  • 拒绝审批后，runtime 必须能继续 task，不是直接把整个 session 搞死。
  • 审批结果必须进入 transcript，保证后续可追溯。
  • 未来如果需要“本轮记住授权”或“当前 session 记住授权”，现在就要在模型里留概念位。

这块决定的不只是安全，也决定 agent 是否“烦人”。 审批策略如果不先定义好，后面体验会非常难修。

Types

核心 Types 草案

这一节不要求现在把每个字段一次性定死，但要求第一版代码必须沿着这些类型边界生长， 不允许 runtime、tools、storage 各自私有一套语义。

RuntimeEvent

type RuntimeEvent = {
  eventId: string;
  sessionId: string;
  taskId?: string;
  timestamp: string;
  source: "user" | "runtime" | "provider" | "tool" | "system";
  type: string;
  payload: unknown;
};

Task

type TaskState =
  | "pending"
  | "running"
  | "waiting_approval"
  | "completed"
  | "failed"
  | "cancelled";

type Task = {
  taskId: string;
  sessionId: string;
  parentTaskId?: string;
  title: string;
  state: TaskState;
  createdAt: string;
  updatedAt: string;
};

ToolDefinition / ToolResult

type ToolApprovalPolicy = "always" | "on-risk" | "never";

type ToolDefinition = {
  name: string;
  description: string;
  inputSchema: unknown;
  approvalPolicy: ToolApprovalPolicy;
};

type ToolResult = {
  ok: boolean;
  summary: string;
  structuredOutput?: unknown;
  rawLogs?: {
    stdout?: string;
    stderr?: string;
  };
  exitCode?: number;
  errorMessage?: string;
};

ApprovalRequest

type ApprovalRequest = {
  approvalId: string;
  sessionId: string;
  taskId?: string;
  toolName: string;
  reason: string;
  risk: "low" | "medium" | "high";
  createdAt: string;
};

SessionRecord

type SessionRecord = {
  sessionId: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  model: string;
  cwd?: string;
};
Boundaries

模块职责边界

  • terminal：只负责输入事件、重绘、布局，不负责业务决策。
  • editor：只负责输入态，不直接调用 provider 或 tools。
  • runtime：唯一有权调度 provider、tools、tasks、approvals。
  • tools：只暴露统一工具接口，不保存 session 语义。
  • providers：只负责模型调用，不感知终端 UI。
  • storage：只保存状态与日志，不发明新的业务流程。
硬规则： 任何 UI 组件都不能直接操作 provider 和 tools；必须经过 runtime。

Priorities

第一阶段要先做的 5 个模块

  1. Editor：多行输入、cursor、history、粘贴、基本快捷键。
  2. Agent Runtime：message、task、streaming、approval、tool lifecycle、runtime events。
  3. Tool Registry：统一工具协议，shell、files 两个最基本工具先打通。
  4. Provider Client：先接一个主 provider，稳定处理 streaming、errors、cancel。
  5. Storage：settings、transcripts、logs、checkpoints 先落地。

Phases

执行阶段

Phase 1：MVP 跑通

  • 单会话 CLI。
  • 稳定输入区。
  • 流式回复。
  • shell tool + file tool。
  • runtime event model 落地。
  • tool contract 落地。
  • transcript 持久化。

完成标准：用户可以稳定输入，模型可以稳定流式返回，shell / file 两个工具可被 agent 调用， 事件和 transcript 能完整记录，出现错误时不会把 session 状态打坏。

Phase 2：Agent 化

  • task list / plan。
  • approval flow。
  • context compaction。
  • background tasks。
  • tool output folding。
  • retry / resume。

完成标准：agent 已经不只是“会聊天”，而是可以管理任务、请求审批、恢复执行，并控制上下文膨胀。

Phase 3：产品化

  • 更强 history / search / jump。
  • 更强 logs 与 diagnostics。
  • workspace restore。
  • 配置体系稳定化。

完成标准：CLI 可以长期使用，错误可诊断，状态可恢复，配置可维护，不再只是工程原型。

Phase 4：再评估桌面端

  • 只有当 CLI 已经成熟且长期可用，再评估是否包一层桌面端。
