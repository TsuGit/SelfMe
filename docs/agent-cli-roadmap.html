Execution Plan

SelfMe Agent CLI Roadmap

当前路线已经收敛：只做一个单体 TypeScript CLI agent，把输入体验、工具执行、审批、安全边界和真实 coding task 闭环做扎实。先不讨论 web、gateway、多端共享后端，也不为未来产品形态提前堆复杂架构。

唯一目标： 把当前 CLI 做成一个真正可用、可持续迭代的 agent 基线，而不是继续扩张产品壳层。
Position

产品边界

  • 这是一个单进程、本地运行的 CLI agent。
  • 当前技术栈固定为 TypeScript，不再讨论 Rust / TypeScript 选型。
  • 重点是 terminal 交互、agent runtime、tool execution、provider integration、local storage。
  • 所有设计都优先服务“真实 coding work 能不能稳定完成”，不是服务多端扩张。
评估标准只有一个：它是否越来越像一个真正能工作的 agent，而不是一个会聊天的终端 UI。
Baseline

当前基线

Terminal

已切到原生终端滚动模型，具备稳定输入区、多行输入、真实光标、流式回复、底部菜单和克制的状态展示。

Commands

当前保留最小命令面：/help、/stop、/read、/write、/edit、/shell。

Approvals

shell、write、edit 已接入风险分级审批；低风险 shell 可直通，高风险 shell 仍需审批，write / edit 已具备任务级续批边界，拒绝后也能稳定回到自然收口。

Storage

已保留 transcript、logs、settings 等必要存储，去掉了复杂恢复和多余状态层。

Runtime Control

已具备单活跃任务模型、忙时拦截新消息、Esc / Ctrl+C / /stop 中断当前任务，以及 provider / shell abort 贯通；同时对可执行请求加入了 action-first 约束，并能把简短确认词解析为“执行上一条已获批准的提议”。

Turn Feedback

已接入整轮级 Working 计时、停止提示和结束收口线，快速完成的轮次只保留弱分隔，不强行展示耗时；同时 runtime 会对 plan-only、progress-only、affirmative follow-up、resume follow-up 和 repeated-result 空转做自动拉回。

Principles

当前阶段的开发原则

  • 先补真实能力，再补展示层修饰。
  • 新功能必须进入统一 runtime 和 tool contract，不允许再长出第二套流程。
  • 临时提示、帮助、命令说明尽量走底部菜单或瞬时提示，不污染主消息流。
  • 所有高风险操作必须有清晰审批，不允许因为“先能用”而绕开。
  • 任何新增复杂度，都必须能直接提升真实任务完成率，否则不做。
Next

下一阶段要做什么

  1. 回到真实任务完成率

    当前不再机械扩 smoke 样本；后续只围绕真实跑出来的问题补回归，例如首轮探索走散、简单失败恢复过长、proposal follow-up 和 resume follow-up 这类 runtime 收敛问题。

  2. 首轮动作更直接

    像“看看项目”“帮我优化下”这类请求，首轮应该更快锁定目标文件和真正的工作对象，减少无价值探索和把“下一步建议”再抛回给用户。

  3. 简单失败恢复更短

    重点继续压 edit 越界、轻量 shell 失败、文件范围误判这类低级失败，让 agent 在同一轮里更短路径修正，而不是白白消耗工具步数预算。

  4. CLI 本体继续打磨

    下一阶段继续回到输入体验、消息结构、命令面、runtime 行为和真实工作流本身；命令系统仍维持最小集，但要继续保证 /stop、审批、帮助、命令补全这些入口稳定可用。

Not Now

明确暂不做

  • web UI。
  • desktop app。
  • gateway / ws / 多端共享后端。
  • 复杂 session browser、history browser、search browser。
  • checkpoint、任务恢复系统、重型计划系统。
  • 为了未来产品化预埋的大而全壳层。
这些不是永远不做，而是当前阶段不允许抢占主线。主线只有一个：把 CLI agent 能力闭环做扎实。
Milestones

阶段里程碑

M1

读文件、跑 shell、审批流稳定；命令面干净；正常交互不污染 transcript。

M2

已补齐写文件和编辑文件工具，agent 可以在审批后真正修改项目内容。

M3

中断控制、单活跃任务模型、整轮级 Working 反馈、按完整轮次裁剪的上下文和长输出收口已接上；同时低风险 shell 直通、任务级续批、发现后 helper 文件续批边界、action-first 约束和分档工具步数预算也已成立。

M4

已接入首批真实 coding task 回归，覆盖成功路径、多步修正、更长链路任务、跨文件协同、嵌套目录与相对导入修复、读取配置后创建 helper 再修现有导入链、单配置源驱动两个已有文件的同步修复、读取配置后创建 shared helper 再修 nested api 入口、读取配置后创建模板文件再修 nested docs 入口、读取配置后创建数据文件再修 nested report 入口、读取配置后先得到成功但不精确的 nested docs 输出再继续收口、先读多个已有文件再对真正问题文件做定向二读和收口、先读多个已有文件再只修真正有问题的 helper 避免过修、长链 coding task 可用扩展预算继续推进、plan-only / progress-only 回复会被拉回继续执行、重复得到相同工具结果时会被强制换动作、中文长链、成功但结果不精确时的继续收口、基于两个已有文件修第三个已有文件、长输出截断后的定向跟进、直接 shell 失败、直接 shell 成功、审批拒绝、wrapped / repaired tool call 解析，以及同任务内读取后发现的 helper 文件续批；这一层基线已经足够支撑下一阶段功能开发。

Definition

什么叫这一版已经成立

  • 用户能稳定输入、编辑、发送多行内容。
  • 模型回复稳定流式输出，工具调用反馈清晰。
  • agent 可以读文件、改文件、跑 shell，并通过审批安全执行。
  • 长一点的对话不会因为上下文和工具日志失控而明显退化。
  • 可以独立完成一类真实小型 coding task，而不是只会执行单点命令。
Immediate

当前建议的直接开工项

下一步直接做 CLI 产品能力开发。基础自动工具调用、风险分级审批、任务级续批、中断控制、整轮状态反馈、长输出收口、真实任务回归和上下文裁剪已经成立， 现在应该继续围绕真实任务完成率推进：优先收首轮动作选择、简单失败恢复链和多步任务连续执行感，而不是再做机械性的测试扩面。
