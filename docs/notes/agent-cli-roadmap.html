Execution Plan

SelfMe Agent CLI Roadmap

当前路线已经收敛：只做一个单体 TypeScript CLI agent，把输入体验、工具执行、审批、安全边界和真实 coding task 闭环做扎实。先不讨论 web、gateway、多端共享后端，也不为未来产品形态提前堆复杂架构。

唯一目标： 把当前 CLI 做成一个真正可用、可持续迭代的 agent 基线，而不是继续扩张产品壳层。
当前开发方法已单独固化在 docs/notes/agent-eval-strategy.html，后续 runtime 和 smoke 推进以那份能力分层、失败分类和退出标准为准。
当前状态说明： 仓库已经具备可运行的 CLI、真实工具调用、审批链路和一批 runtime 回归；stage summary / progress-only reply 之后的 continuation、resume 和多步骤执行基线已经进入回归保护，显式点名项目根或文件的首轮 direct-start 也已经进入 smoke 覆盖，pnpm smoke:agent 当前是全绿状态。接下来的重点不再是补 smoke 闭环本身，而是把这套收敛能力继续压实到真实交互场景里。
Position

产品边界

  • 这是一个单进程、本地运行的 CLI agent。
  • 当前技术栈固定为 TypeScript，不再讨论 Rust / TypeScript 选型。
  • 重点是 terminal 交互、agent runtime、tool execution、provider integration、workspace state storage。
  • 所有设计都优先服务“真实 coding work 能不能稳定完成”，不是服务多端扩张。
评估标准只有一个：它是否越来越像一个真正能工作的 agent，而不是一个会聊天的终端 UI。
Baseline

当前基线

Terminal

已切到原生终端滚动模型，具备稳定输入区、多行输入、真实光标、流式回复、底部菜单和克制的状态展示。

Commands

当前保留最小命令面：/help、/stop、/read、/write、/edit、/shell。

Approvals

shell、write、edit 已接入风险分级审批；低风险 shell 可直通，高风险 shell 仍需审批，write / edit 已具备任务级续批边界；当会话里只有一个待审批项时，用户可直接用“可以 / 拒绝 / 继续 / 同意”这类自然语言完成审批，多待审批并存时仍强制走带 id 的精确批准，而且模糊的自然语言批准不会偷偷消耗任何一个待审批项；审批通过后，原任务会在同一轮里继续跑后续 write / edit / verify，不会因为等过一次审批就断成两截；审批等待留在底部面板，审批结果会作为独立消息块落历史，不再和普通任务流紧贴混排。

Storage

已保留 transcript、logs、settings 等必要存储，去掉了复杂恢复和多余状态层。

Runtime Control

已具备单活跃任务模型、忙时拦截新消息、Esc / Ctrl+C / /stop 中断当前任务，以及 provider / shell abort 贯通；同时对可执行请求加入了 action-first 约束，并能把简短确认词解析为“执行上一条已获批准的提议”或“继续最近未完成任务”；像 improve / optimize / 优化 / 重构 这类动作型请求，也会进入同一条执行收敛链，而不是停在建议语句；即便用户没显式写 verify，这类真正的修改任务在读完文件后如果模型只给分析、说明或多余确认问题，runtime 也会继续把它拉回到具体 edit / write；如果用户明确要求同一轮里改多个具体目标，runtime 也不会在第一处成功 edit / write 后，或者在只覆盖了第一处改动的完成口吻后，就提早收尾；这条收敛现在也覆盖“第一处 edit 成功后反问要不要继续改第二个文件”这种低价值停顿；超过三处显式文件目标的项目级改造现在也会自动拿到扩展 tool-step 预算，不再只能靠 verify / exact-output 关键词触发长链执行额度。

Turn Feedback

已接入整轮级 Working 计时、停止提示和结束收口线，快速完成的轮次只保留弱分隔，不强行展示耗时；阶段性 assistant 总结会先落历史再继续干活，不再被后续 Working 覆盖；即便阶段性总结里已经提到“已创建 helper，下一步我会修 runner 并验证”这类完成词，runtime 也会把它识别为中途 stage 而不是最终完成；同时 runtime 会对 plan-only、progress-only、affirmative follow-up、resume follow-up、过薄 follow-up 收尾和 repeated-result 空转做自动拉回，并会在可执行任务里统一裁掉“可以 / 已继续 / 好的”这类低价值起手词，避免结果回复继续带明显确认腔；当验证链里的失败点已经前移到新的 helper / 文件时，后续续跑也会优先跟着最新失败点走，而不是倒退回更早那一步；如果任务在已有失败点和已有工作文件上下文里被 stop，中断后一句“继续”也会尽量直接接回那个修复点，而不是重做更早的创建步骤，这条恢复能力现在也覆盖项目级多文件 + verify 链，而不只是小文件修复；如果 stop 发生在待审批动作上，resume 提示里也会显式带回最近工作文件、最近 tool 结果和那条被打断的审批动作，尽量把模型直接拉回待执行 edit / write，而不是重读项目入口；如果最近确实有一个 interrupted task，而不是一个待执行 proposal，那么 bare `继续` 甚至 bare `可以` 现在也会优先走 resume，而不是被误当成普通确认腔；即便最早的任务本来是从 assistant proposal 开始执行的，只要 proposal 已经实际跑起来又在中途被打断，后续 bare `可以` 也会优先恢复已开跑的执行链，而不是回到 proposal 起点重新审批式重跑；如果 stop 发生在项目级 stage summary 刚落完、下一步还没执行时，后续一句“继续”也会优先接住那条已经收窄好的下一步，而不是重新从 listing、package 或 app.js 大入口起跑；如果任务刚好卡在 tool-step ceiling 上，runtime 现在也会把那条尚未执行的具体文件目标记成 pending next step，后续 resume 会优先回到这个被上限截断的位置，而不是退回更宽的项目入口或旧的读取链；如果模型在任务中途已经读到明确工作文件却突然反问“要不要我改这个文件”，runtime 也会把这种无效反问拉回到具体 edit，而不是把它当成真的阻塞；这条拉回现在也覆盖项目级 verify / exact-output 链里的 shell near-miss 阶段，所以像 app-only 之后的“要不要我继续改 views”也不会把整条执行链切断；如果任务已经精确完成，模型后面再多发一轮无意义 tool call，runtime 也会直接压回最终收口，而不是继续做多余验证；如果最终回复虽然是完成态，但还在长篇复述中途失败和修复过程、短句回顾 earlier / failed 这类旧失败点，或把无关旧文件重新带回收尾，runtime 也会把它收紧成更短、更聚焦结果的结束语。

Project Inspection

像“看看项目”这类请求，agent 现在会在目录 listing 之后优先继续读取最像项目入口的 package.json、README 或主源文件，而不是停在泛泛追问；如果用户一开始就明确点了具体项目根或具体文件，例如 node-todo、packages/demo-core、node-todo/app.js，runtime 也会在首轮 prompt 里显式注入 preferred starting target，尽量直接从那个项目入口或文件起跑，而不是先做一轮无价值的 workspace listing；如果用户明确要看“整个项目”而不是只看入口，runtime 也会在读完项目入口后继续下钻到核心实现文件，而且不会在第一个入口实现文件就急着收口；如果用户语义已经带有“帮我优化下”这类改造意图，读到项目入口后还会继续落到更像工作对象的实现文件，而不是停在“下一步我可以看 app.js”这种半路说明；如果同一轮请求里已经明确点了多个具体文件，runtime 也会在第一处成功改动后继续推进到下一处具体文件，而不是把“我下一步会继续改 views”这种阶段性说明误当成最终完成；如果用户先看完项目、下一句才单独说“帮我优化下”或“你能帮我重新写个项目吗”，runtime 也会优先锚到最近刚 inspected 的项目/文件上下文，而不是把它当成完全无锚点的新问题；如果前一轮 assistant 已经给出明确的项目级 rewrite proposal，后面再来一句宽泛的“重新写个项目”，runtime 也会直接执行那套跨文件 proposal，而不是缩水成单文件试探；如果前一轮 assistant 已经给出明确的项目级 optimize proposal，后面再来一句宽泛的“帮我优化下”，runtime 也会继续落回那条项目级执行上下文，而不是退回泛化建议；如果普通项目执行链本身已经跑到一半又被 stop，后续一句宽泛的“帮我优化下”或“帮我看看”，现在也会优先恢复那条中断执行链，而不是重新回到 listing、package 或泛化分析；如果这套 rewrite proposal 本身还带有 verify / exact-output 约束，后续宽 follow-up 也会继续把整条改动和验证链跑完，而不是只把 proposal 执行到 edit 阶段就停住；如果这套 rewrite proposal 已经实际执行到一半又被 stop，后续 bare `可以`、宽泛的“重新写个项目”，以及对应的 optimize / inspect follow-up，runtime 也会优先恢复那条已经开跑的执行链，而不是回到 proposal 起点重读和重改前半段；如果用户先要 rewrite 方案、后面再用一句“可以”批准，runtime 也会把这句批准绑定回刚才的项目级提议并直接开始跨文件执行，而不是重新回到只读或解释态；如果最近已经有明确的 editable working file，宽 follow-up 会优先直接回到这个文件，而不是倒退去重读更宽的项目入口；这条锚点策略现在不仅覆盖优化类 follow-up，也覆盖“帮我看看 / 看看这个”这类 inspect follow-up。

Checkpoint

当前恢复锚点

  • 截至 2026-07-07，npm 包名已经定为 selfme，安装命令是 npm i -g selfme，可执行命令是 selfme。
  • 当前仓库的终端交互、审批、工具消息和基础恢复链已经够用，后续主线不再回到“先补产品壳层”或“先扩表面功能”。
  • pnpm smoke:agent 已经覆盖 continuation、resume、project inspection、rewrite / optimize follow-up、exact-output / verify 收口等关键 runtime 收敛路径，并保持全绿。
  • 除了纯 runtime fixture，当前 smoke 还补进了 terminal loop 端到端回归：真实键入的多步 prompt 会穿过 stdin -> editor -> terminal loop -> runtime 整条链，并在同一任务里跨过 stage summary 继续落到剩余文件 edit；同一条覆盖现在还包含 Esc 中断后再键入“还能继续吗”的恢复链，要求 resume 直接接回 pending command / pending file，而不是重新翻项目入口；另外也补进了 approval-wait 分支，要求任务即便是在待审批 edit 上被键盘中断，后续 typed resume 也要直接回到那条 pending approval 对应的 edit，而不是重读 package / app / view；这条现在也覆盖更真实的子分支：如果第一处 edit 已经完成、stage summary 已经把任务收窄到第二个文件，而 stop 正好发生在第二次审批等待中，resume 仍然要直接接回那条挂起 edit；同时，approval 面板打开时直接键入 可以 / 不可以 这类自然语言审批回复，也已经进入 terminal-loop 回归保护，不再只依赖面板选项或 /approve <id>；另外，typed 长任务跨过 per-slice tool-step ceiling 后自动续跑的真实终端链路也已进回归，要求 runtime 直接接着跑 pending next step，而不是把 step-limit 旧硬停重新暴露给用户；现在还额外覆盖了 execution-convergence 分支，也就是模型在真实 failed verification 之后先发一段 explanation-only reply，runtime 仍要在同一任务里回到已知 working file、完成 edit 并重跑验证。
  • 如果用户首轮就明确点名 node-todo、packages/demo-core 这类项目根，或 node-todo/app.js 这类具体文件，runtime 现在会优先从该项目入口或该文件起跑，而不是先做一次工作区 listing；这条也覆盖更短的 prompt 形态，例如 看看 node-todo、直接优化 node-todo、直接重写 packages/demo-core，不再要求用户必须显式写出“项目 / project”。
  • 普通工具执行失败（例如缺文件、edit range 越界）现在也会进入统一的 failed tool result 历史，而不是掉进零散的 runtime error 分支，后续 continuation / repair 能直接复用这条失败锚点。
  • 项目入口、第一个核心实现文件，或第一处已落到的工作文件后的长解释句，现在也会继续被拉回到下一个真正动作，而不是停在“我已经看出问题了”的说明态。
  • 即便 stop 发生在项目入口刚读完、assistant 还没来得及吐出“下一步我会继续看 app.js / server.js”这类阶段总结之前，resume 现在也会优先根据最近 tool 上下文推断下一步，而不是重新从 listing、package 或 README 起跑。
  • 如果任务不是被用户手动 stop，而是刚好撞到 tool-step 上限，runtime 现在也会先把那条尚未执行的 files / edit / write 目标记录成 pending next step；这条恢复不只覆盖“下一步还没读文件”，也覆盖“目标文件已经读到了，但真正的 edit 还没来得及执行就被上限截断”的长任务中段；如果撞到的是 assistant-pass 上限，或者 repeated assistant stall / repeated tool stall 这类已经收窄到具体文件的循环失败，continuation prompt 里明确的目标文件也会被记下来；即便卡住点不是 shell / files 重复，而是连续 malformed tool call、unknown tool 或 invalid tool input 这种模型工具调用循环，只要前面已经有明确工作目标，甚至目标还只是来自 original request、还没跑出第一条真实 tool result，后续一句 继续 也会优先接回那个文件，而不是回到更宽的目录勘查重新起跑。
  • 上下文裁剪后的 long-history proposal approval 现在不只覆盖 nested packages/... 场景，也覆盖普通顶层项目 rewrite proposal；像 node-todo 这种更贴近真实对话的项目，在长会话里用户一句 可以 之后也能继续锚到刚才的 proposal 和 working file，而不是漂移到旁边的 decoy 项目。
  • runtime prompt 现在明确要求 concrete coding task 倾向连续完成下一个短链的 2 到 4 步；同样的收紧也落到了 approved proposal、execution-stage explanation 和 actionable failure continuation 上，用来减少“做一步就解释”或“失败一句就收尾”的停顿感。
  • 下次继续时，优先看真实交互里是否仍然存在“一次工具、一句收尾、提前停住”的体感问题，而不是先回头补更多 smoke 文案分支。
  • 如果需要快速判断当前是不是跑偏，先跑 pnpm smoke:agent 守住基线，再去复现真实会话里的 continuation / resume / long-task 行为。
下次继续的默认顺序： 先读这个 checkpoint，先用 pnpm smoke:agent 守住已收敛基线，再复现真实会话里的执行连续性问题；后续主线只修 runtime 收敛和执行稳定性，不扩新表面。
Principles

当前阶段的开发原则

  • 先补真实能力，再补展示层修饰。
  • 新功能必须进入统一 runtime 和 tool contract，不允许再长出第二套流程。
  • 临时提示、帮助、命令说明尽量走底部菜单或瞬时提示，不污染主消息流。
  • 所有高风险操作必须有清晰审批，不允许因为“先能用”而绕开。
  • 任何新增复杂度，都必须能直接提升真实任务完成率，否则不做。
Next

下一阶段要做什么

  1. 回到真实任务完成率

    当前不再机械扩 smoke 样本；后续只围绕真实跑出来的问题补回归，例如真实终端里的连续执行停顿、恢复链过长、目录勘查后重复探索、以及收尾语义不稳这类 runtime 收敛问题。

  2. 首轮动作更直接

    像“看看项目”“帮我优化下”这类请求，首轮应该更快锁定目标文件和真正的工作对象；这一层已经不再满足于只列项目候选，下一步要继续减少无价值探索和重复 listing。

  3. 多步连续执行更自然

    下一阶段重点不再是单点回复样式，而是让 agent 在同一任务里更自然地连续做 2 到 4 步：读、判、改、验之间少停顿，减少“一步一停、一步一答”的割裂感。

  4. CLI 本体继续打磨

    输入体验和消息结构当前已经够用，下一阶段继续聚焦 runtime 行为、审批策略和真实工作流本身；命令系统仍维持最小集，但要继续保证 /stop、审批、帮助、命令补全这些入口稳定可用。

Not Now

明确暂不做

  • web UI。
  • desktop app。
  • gateway / ws / 多端共享后端。
  • 复杂 session browser、history browser、search browser。
  • checkpoint、任务恢复系统、重型计划系统。
  • 为了未来产品化预埋的大而全壳层。
这些不是永远不做，而是当前阶段不允许抢占主线。主线只有一个：把 CLI agent 能力闭环做扎实。

  • 模型回复稳定流式输出，工具调用反馈清晰。

  • 长一点的对话不会因为上下文和工具日志失控而明显退化。
  • 可以独立完成一类真实小型 coding task，而不是只会执行单点命令。
