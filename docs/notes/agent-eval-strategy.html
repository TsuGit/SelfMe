SelfMe Agent Eval Strategy

这份文档定义 SelfMe CLI agent 的开发方法，不讨论产品愿景，只约束日常开发怎么推进、怎么判断有效、以及什么问题必须先解决。

核心原则： 不是先堆功能，而是先提高真实任务完成率；不是按感觉修，而是把失败沉淀成可重复的回归。
Order

能力分层

L1 执行闭环

任务能开工，工具能调用，审批不掉链，中断可控，读写文件和 shell 能形成最基本的执行回路。

L2 收敛能力

失败后继续修，成功但不精确时继续收口，中途 explanation / question / proposal 不会把任务打断。

L3 项目级任务

项目勘查后能落到真正工作文件，多文件改造不会提前收尾，能跨文件持续推进到验证结束。

L4 长任务稳定性

上下文压缩、恢复、长输出、失败点前移、working file anchor 和长链连续执行在真实任务里稳定成立。

只有当前一层达到退出标准，下一层才允许成为主线。允许提前探索，但不允许让探索反客为主。
Taxonomy

失败分类

T1 不开工

用户已明确给出任务，但 agent 仍停在计划、解释、确认或泛泛提议，没有进入第一步真实工具执行。

T2 一步一停

agent 每做一步就回一句，不能自然串起读、判、改、验，导致中小任务也被切成零散轮次。

T3 失败后中断

验证失败、读文件失败或工具失败后，agent 只复述错误，不沿最新失败点继续修复。

T4 近似成功后停住

脚本已经跑通，但结果不精确，agent 却把任务当成完成，或者改成反问、长提议、重新讨论。

T5 项目级跑偏

看完项目后停在入口文件、重复 listing、重读宽入口，或者只改第一处目标就提前结束。

T6 终态回复失焦

任务已完成，但回复仍在长篇复盘旧失败、引用无关旧文件、抛额外工具调用或继续问无效问题。

T7 审批断链

审批通过后任务不能在同一条链路里继续，或多待审批并存时自然语言批准误命中错误对象。

T8 恢复失真

中断后“继续”不能回到最近失败点、最近工作文件或最近验证目标，而是倒退到更早阶段。

Loop

开发循环

  1. 先收真实失败样本，不先猜。
  2. 把失败归类到一个明确 taxonomy，不允许“什么都像一点”。
  3. 先写最小 smoke 回归，再决定 runtime / prompt / tool contract 该改哪层。
  4. 只做刚好能让该类失败稳定消失的最小修正，不顺手扩功能。
  5. 跑全量 tsc 和 pnpm smoke:agent，确认没有破坏别的类。
  6. 同步文档，把这条能力写进基线或阶段说明。
如果一个问题不能先变成回归，它就还不是一个被控制住的工程问题，只是一次现场观察。
Eval

评测策略

  • 单链回归：覆盖单文件修复、单命令验证、审批直通、中断恢复等最小闭环。
  • 收敛回归：覆盖 explanation-only、blocking question、broad proposal、completion-sounding reply、repeated result 等中途偏航。
  • 项目级回归：覆盖“看看项目”“帮我优化”“直接重写/直接改造”这类宽任务，从 listing 到工作文件再到多文件改造。
  • 终态回归：覆盖结果收紧、禁止旧失败复盘、禁止无关旧文件漂移、禁止成功后额外工具调用。
  • 人工手测：只验证 smoke 不容易表达的真实交互质感，不替代 smoke。
Exit

阶段退出标准

S1 执行闭环完成

能稳定开工、调用工具、处理审批和中断，且 smoke 覆盖基本失败链。

S2 收敛能力完成

中途 explanation、question、proposal、near-miss、重复结果都不会让任务脱轨。

S3 项目任务完成

项目勘查、项目优化、多文件改造、明确授权后的直接开工，都有真实回归覆盖并能稳定通过。

S4 长任务完成

长输出、上下文压缩、恢复链、最新失败点和工作文件锚点，在真实任务里不明显退化。

Immediate

当前主线

  1. 继续补 T2 / T4 / T5 相关回归，也就是连续执行、近似成功继续收口、项目级宽任务不跑偏。
  2. 把“看看项目 -> 直接改造/重写 -> 明确同意 -> 必须真正开工”做成项目级 smoke，而不是只测单文件修复。
  3. 补多待审批并存时的自然语言审批边界，确保不会误批错误对象。
